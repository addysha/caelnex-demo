Phase 0 · Founding architecture

A configuration graph that watches the seams between your systems.

Caelnex is a graph-native record of decisions, builds, and certifications for launch organisations. It does not replace your MES, PLM, ERP, or QMS. It watches the space between them — where the most expensive failures in this industry's history have occurred.

Rule 0. If there is no time left for a human to make the call, the system has already failed — earlier, upstream, and silently. Caelnex's job is not to make decisions. It is to make sure the people accountable for a decision still have time to make it.
System architecture — five layers

Raw telemetry never reaches the graph.

Sensor data and the configuration graph are deliberately separated. The graph stores verdicts, decisions, and relationships — never raw measurements.

LAYER 5

Operational outputs

Readiness board, go/no-go poll, audit package, anomaly alerts — the views people actually work from.

LAYER 4

Configuration graph

Neo4j. Stores component, specification, build, certification, and decision nodes — plus the relationships between them. Forward-only status transitions. Nothing is ever deleted.

LAYER 3

Organ 7 — deterministic gate

Hard-coded validation rules only. No probabilistic AI at this layer. An advisory AI layer can extend warning time — it can never override a pass or fail.

LAYER 2

Time-series + edge processing

Raw sensor telemetry lives here — TimescaleDB, processed at the edge. Only high-level verdicts are pushed up to the graph.

LAYER 1

Physical root of trust

Smart torque tools, RFID-tagged components, instruments that report their own readings — capture, not data entry.

Seven organs, one graph

Each organ is a view. The graph is the truth.

Engineering, software, manufacturing, supply chain, launch operations, and compliance all read and write to the same underlying graph — with Organ 7 validating every change before it lands.

O1

Design & Engineering

Change propagation, revision genealogy, decision nodes that carry the reasoning behind a change — not just the change itself.

O2

Flight Software

Every build is validated against the specific hardware configuration it was tested with. A build with no validated pairing cannot fly.

O3

Manufacturing & Integration

The digital traveller. Sequence enforcement, vehicle serial genealogy, and tool-submitted values — not technician-entered ones.

O4

Supply Chain

Part genealogy from supplier to installation. Cross-batch certification analysis that surfaces statistical patterns no single certificate shows.

O5

Launch Operations

A live readiness score computed from the graph itself — not assembled from status meetings. Formal, attributed go/no-go votes.

O6

Compliance & Safety

ITAR-aware classification from the moment a node is created. Cryptographically chained audit trail, exportable on demand.

O7

Compliance Index Engine

The pre-submission gate every other organ writes through. Five deterministic rules — unit completeness, hardware pairing, tolerance bounds, escalation deadlines, and certification clustering — each calibrated under Rule 0 to preserve decision time, not just catch errors.

Why these rules exist

Three documented failures. $1.4B combined.

Every deterministic rule in Organ 7 traces to a specific, investigated failure — not a hypothetical one.

IncidentYearLossRoot causeOrgan 7 rule
Mars Climate Orbiter 1999 $327.6M Ground software output thrust in pound-force seconds; navigation expected newton-seconds. Units field was effectively undefined. O7-SPEC-001 — unit field mandatory
Ariane 5 Flight 501 1996 $370M Reused Ariane 4 inertial reference software, never validated against Ariane 5's trajectory profile. O7-BUILD-001 — validated hardware pairing required
NASA OCO & Glory (SPI fraud) 2009 / 2011 $697M Falsified aluminium certifications clustered at the specification boundary across 19 years and hundreds of batches. O7-CERT-001 — boundary clustering detection
